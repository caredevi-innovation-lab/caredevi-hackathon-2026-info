## Model Choices and Rationale

### Primary LLM: Llama3-OpenBioLLM-70B

**Why this model:** OpenBioLLM is fine-tuned on biomedical and clinical corpora, making it significantly more calibrated for healthcare terminology, clinical reasoning, and appropriate hedging language than general-purpose models. It produces language like "consistent with" and "suspected" rather than overconfident diagnostic claims.

**How it is used:** Synthesis only. The LLM receives a structured prompt containing:
- Extracted symptoms (from deterministic NER, not raw input)
- Validated sensor readings
- Relevant clinical protocols (retrieved via RAG)
- FHIR patient history
- Deterministic detection results and ESI level

The LLM is explicitly instructed to answer only from provided context, use hedged language, flag conflicts, and respond in structured JSON. It does not drive detection, ESI scoring, or hospital routing — those are all deterministic.

### Fallback LLM: Qwen2.5-Coder-32B-Instruct

Used when the primary model is unavailable. Produces the same JSON schema. If both LLMs fail, the system falls back entirely to deterministic outputs with no LLM involvement.

### Clinical NER: scispaCy `en_core_sci_md`

A biomedical spaCy model trained on scientific literature. Used to extract symptoms, medications, and clinical findings from voice transcripts and free-text input. Falls back to keyword matching if the model is unavailable.

### RAG: ChromaDB + sentence-transformers/all-MiniLM-L6-v2

A lightweight, local embedding model used to retrieve relevant clinical protocols at query time. This grounds the LLM in authoritative guidelines rather than relying on parametric knowledge, which reduces hallucination risk.

## Bias Considerations

### Training Data Bias
- **Llama3-OpenBioLLM-70B** was trained on biomedical literature that historically underrepresents certain demographics (women in cardiac research, racial minorities in clinical trials). This can lead to lower confidence scores or missed signals for presentations that deviate from the historically documented "textbook" case.
- **Mitigation:** Critical detection (cardiac arrest, STEMI, stroke, anaphylaxis) is handled by deterministic rules, not the LLM. The LLM is only used for differential synthesis. A missed pattern by the LLM does not suppress an alert if the rule engine fires.

### Symptom Presentation Bias
- Cardiac events present differently in women, diabetics, and elderly patients (atypical symptoms: nausea, jaw pain, fatigue without classic chest pain). The rule-based detectors partially account for this (e.g., `cardiac_hx` factor, radiation patterns) but cannot cover all atypical presentations.
- **Mitigation:** The ESI scoring system defaults to ESI 3 (urgent) when uncertainty is high, erring on the side of faster response rather than dismissal.

### Geographic Bias in Hospital Routing
- Hospital data (`hospitals.json`) is currently synthetic. In production, hospital capability data would need to be continuously updated. Stale data could route to a hospital lacking required capabilities.
- **Mitigation:** The routing module includes a `fallback_triggered` flag and `re_evaluation_triggers` that alert paramedics if the best available hospital is missing required capabilities.

### Language and Accent Bias
- Voice transcript quality directly affects NER accuracy. Accented speech, non-native English speakers, or speech under duress may produce lower-quality transcripts.
- **Mitigation:** The `detect_speech_quality` function flags aphasia, slurred speech, and incoherence as clinical signals rather than dismissing them. The system raises, not lowers, concern when speech quality is poor.

## Failure Cases and Mitigations

| Failure Case | Risk | Mitigation |
|---|---|---|
| **LLM hallucination** — model invents drug interactions or lab values not in context | Incorrect differential could mislead paramedics | LLM is constrained by RAG context; system prompt explicitly prohibits inventing data; deterministic detection runs independently and is always shown |
| **Sensor spoofing / hardware failure** — wearable reports implausible values | False positive or suppressed alert | `collide_and_resolve` cross-validates multiple sensors; `SENSOR_ERROR` verdict triggers fallback alert mode |
| **Both LLMs unavailable** — API outage or network failure | No LLM-generated differential | Fully deterministic fallback produces ESI, routing, and FHIR bundle — no degradation to safety-critical outputs |
| **Unknown condition** — presentation doesn't match any detection rule | No alert fires; patient triaged as ESI 3 | The system defaults to `er_basic + icu` routing for unknown presentations; "also_call_911" flag is always true |
| **Stale patient FHIR data** — patient's medication list is out of date | Drug interactions or allergies missed | System marks FHIR-sourced data with the bundle's `lastUpdated` timestamp; paramedics are instructed to verify on arrival |
| **Overfitting on ESI** — confident high-ESI prediction delays departure for lower-acuity patient | Ambulance sent unnecessarily | ESI scoring errs toward lower numbers (higher urgency). ESI 3 is the floor — the system never says a patient is fine |
| **False positive cardiac arrest** — HR=0 in sensor due to watch removal | Unnecessary 911 dispatch | `watch_on_wrist` flag required for cardiac arrest detection; multi-signal agreement required |
| **Routing to wrong hospital** — capability data incorrect or outdated | Patient arrives at hospital lacking required capabilities | `fallback_triggered` flag propagated to UI; paramedics explicitly warned; `re_evaluation_triggers` listed in response |

## Safety Design Principles

1. **Deterministic-first:** Detection, ESI, and routing never depend on the LLM. LLM failures silently fall back.
2. **Always call 911:** Every API response includes `"also_call_911": true`. The system augments, never replaces, emergency services.
3. **Mandatory disclaimer:** Every LLM output includes: *"This is AI-generated preliminary information. Not a clinical diagnosis."* This appears in the FHIR bundle, the handoff summary, and the UI.
4. **Confidence as uncertainty, not authority:** The model is instructed to use hedged language. Confidence scores represent plausibility, not clinical certainty.
5. **No PHI:** All patient data is synthetic (Synthea). No real patient identifiers are stored, logged, or transmitted.
6. **Conflict surfacing:** Conflicting sensor readings are flagged explicitly in the response and the FHIR bundle rather than silently resolved.

## Regulatory Awareness

This prototype is not FDA-cleared and is not a Software as a Medical Device (SaMD) submission. In a production path toward clinical deployment, it would require:

- FDA 510(k) clearance or De Novo classification as a clinical decision support tool
- HIPAA Business Associate Agreement with any hospital system receiving FHIR bundles
- Prospective clinical validation across demographically diverse patient populations
- Ongoing bias monitoring across age, sex, race, and comorbidity subgroups
- Explainability audit trail for every routing and ESI decision

These requirements are acknowledged and would be addressed before any real-world deployment.
